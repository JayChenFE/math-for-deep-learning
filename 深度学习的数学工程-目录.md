20.2　float32 与 float64：精度导致的 `(1+1e-8) - 1 = 0`

20.3　**float32 vs float16 vs bfloat16**

- fp16：范围小（最大值 65504），容易溢出→用于推理加速
- bf16：范围和 fp32 一样大，但精度更低（7 位有效数字）→训练的"安全加速器"

## 第24章　优化算法：从 SGD 到 AdamW —— 训练加速的工程智慧

24.1　SGD 回顾：`θ -= lr * ∇L(θ)`（纯梯度下降）

24.2　**SGD + Momentum**：给梯度加"惯性"——小球滚下山

- 物理直觉：下山的小球有动量，不会在每个坑里停下来
- 代码：对比纯 SGD 和 Momentum 在狭长峡谷函数（f(x,y)=x²+10y²）上的收敛轨迹

  24.3　**Adam**：自适应学习率

- 直觉 1（动量项）：稀疏参数大步走，活跃参数小步走
- 直觉 2（自适应项）：梯度方差大的方向走小步（谨慎），方差小的方向走大步（加速）
- **Adam 是训练 Transformer 的事实标准** —— 不理解它就无法 debug 训练曲线

  24.4　**AdamW**：把 Weight Decay 从 L2 正则化中"解耦"

- Weight Decay = 每次更新前先让参数缩小一点点：`θ *= (1 - lr * λ)`
- 与 L2 正则化在 SGD 下等价，在 Adam 下不等价 → AdamW 修复了这个 bug
- HuggingFace Trainer 默认用的就是 AdamW

  24.5　**学习率调度**：为什么 Transformer 训练前几步要小学习率？

- Warmup：从 0 线性增长到目标 lr（前 4000 步）——防止训练初期不稳定
- Cosine Decay：从目标 lr 余弦衰减到 0——平滑收束
- 代码：三种调度策略（constant、step、cosine）的 loss 曲线对比

  24.6　**Gradient Accumulation**：batch_size 不够大时的"攒批技巧"

- 代码：每 4 步才调一次 `optimizer.step()`，等效 batch_size 扩大 4 倍

  24.7　**Agent 微调中的学习率策略** 🆕——ReAct Agent 训练通常分两阶段：**指令微调**阶段用 Warmup + 余弦衰减（标准配置），**RLHF 阶段**用极低学习率（1e-6）防止灾难性遗忘。代码：在同一个模型上对比分阶段 LR 与恒定 LR 的 loss 曲线差异。

## 第29章　注意力核心：QKV与缩放点积

> **本章目标**：手写单头自注意力，彻底搞懂 `Q @ Kᵀ / √d` 的物理意义。不碰多头，不碰FFN。

> 🕵️ **开篇导语：一场“信息审讯”**
>
> 想象你是一位侦探，正坐在审讯室里，面对一排嫌疑人（**输入序列**）。此刻，你的目光紧紧盯着**当前**这个人（**Query**）——你心里有一个明确的问题：“昨晚10点你在哪儿？”
>
> 你迅速扫视屋内其他所有人，他们每人胸前都挂着一张“线索牌”（**Keys**）：有人写“我在酒吧”，有人写“我在家”，还有人写“我看见了可疑车辆”。你的大脑飞速计算：谁的牌子和我的问题最匹配（**Q·K 点积**）？匹配度越高，你就越应该集中精力听取**这个人**的完整证词（**Values**）；匹配度太低的人，你直接无视（Softmax后权重趋近于0）。这就是**缩放点积注意力**最底层的物理直觉。
>
> 但仅凭“语义匹配”还不够——嫌疑人的供词必须遵循时间顺序（你不可能让昨晚10点发生的事，被今早8点的证词提前透露）。这就是**因果掩码（Causal Mask）**的职责：屏蔽未来信息。同时，为了让模型感知“第3个词”和“第8个词”的相对位置，我们还必须给每个嫌疑人贴上一张**位置标签**（Sinusoidal、Learned，或当前最流行的 **RoPE旋转位置编码**）。
>
> 本章，我们不碰多头，不碰FFN——只专注一件事：把这场“信息审讯”逐行翻译成NumPy/PyTorch张量代码。当你读完本章，`Q @ K.T / sqrt(d_k)` 在你眼中将不再是一堆抽象符号，而是嫌疑人胸前那张正在闪烁的“匹配度评分表”。

29.1　**从词向量到嵌入**：Token → 查表 → 向量（链接第 4 章加法），代码用随机矩阵模拟 Embedding 层，`X.shape = (Batch, SeqLen, d_model)`

29.2　**位置编码的三种直觉**（呼应第 4 章向量加法）

- Sinusoidal（原始）：用 sin/cos 给位置打标签，无需额外参数
- Learned（BERT）：可学习，但定长有损
- **RoPE 直觉版**（LLaMA/Qwen）：旋转位置编码——把位置信息“旋”进向量里（只讲“为什么旋转能表达相对位置”，复数推导留给附录 G）

  29.3　**QKV 三兄弟的角色扮演**（核心类比）

- **Q（Query）**：“我要找什么？”（当前词的需求）
- **K（Key）**：“我有什么？”（所有词的标签）
- **V（Value）**：“找到后取出什么？”（实际要传递的内容）
- 代码：`Q = X @ W_Q`, `K = X @ W_K`, `V = X @ W_V`（强调第 6 章矩阵乘法）

> 陪审团给出初步意见后，信息会被送入**分析室（FFN）**做深度推理——这个两层线性层占掉整个 Block 三分之二的参数量，是模型真正“存储世界知识”的部位。而为了防止层层传递导致信息失真，我们修了一条**残差高速公路**（`x + Sublayer(x)`），配合 **Pre-Norm** 确保梯度在反向传播时能一路畅行无阻直达底层。

## 第32章　KV Cache的形状芭蕾 —— 推理加速的物理内核

> **本章目标**：彻底撕开 KV Cache 的面具，让读者闭着眼都能画出带 Cache 的注意力形状流。

> 🕵️ **开篇导语：档案室的“索引革命”**
>
> 案件复盘室里，设备已经练就了一身破案本领。但现在，它被推到了 **一线审讯现场（推理模式）** 。嫌疑人每说一个字，设备就要立刻判断：这是不是关键证据？它必须 **自回归（Autoregressive）** 地处理：听完第一个字，判断；听完第二个字，判断；听完第三个字，判断……直到嫌疑人把话说完。
>
> 最笨的做法是：每听到一个新字，就把前面所有字连起来重新读一遍。第一个字时读 1 个字，第二个字时读 2 个字，第 100 个字时读 100 个字——总阅读量是 `O(N²)`。**这正是原始 Attention 在推理时的致命伤。**
>
> 聪明的探员迅速建立了一间 **档案室（KV Cache）**。第一次阅读整份笔录（Prefill 阶段）时，他把每个位置的 **K（线索标签）** 和 **V（证词内容）** 归档入册。此后每听到一个新字，他只计算这个字的 **Q（当前疑问）** ，然后去档案室调取所有历史 K/V 做匹配——**历史内容永不重复计算**。
>
> 档案室里的案卷形状极其规整：`(B, H, T, Dh)`。新来的 Q 形状是 `(B, H, 1, Dh)`，它与历史 K `(B, H, T, Dh)` 做矩阵乘法，得到注意力分数 `(B, H, 1, T)`。**用具体数字走一遍**：`B=2, H=8, T=1024, Dh=64` → `(2,8,1,64) @ (2,8,64,1024) = (2,8,1,1024)`。这一个形状变化，把推理复杂度从 `O(N²)` 降到了 `O(N)`。
>
> 但档案室也有它的物理极限。当卷宗积累到 2048 份（SeqLen=2048），占用的书架空间（显存）用公式 `2 × L × H × Dh × 字节数 × B` 精确可算。LLaMA-7B 推理时，**档案室吃掉了一半以上的显存**。于是有人发明了 **PagedAttention**：把案卷切成“页”，像操作系统的虚拟内存一样灵活调度——这是 vLLM 的底层秘密。

## 第33章　解码策略的“人格分裂”实验 —— 从贪心到核采样

> **本章目标**：不只讲采样公式，而是带着读者在“输出质量”和“多样性”之间做工程调参实验。

> 🕵️ **开篇导语：审讯策略的“人格分裂”实验**
>
> 设备推理飞快，但它面临一个更微妙的问题：**每次只输出一个 Token，该选哪个？**
>
> 如果永远选概率最高的那个（**Greedy 策略**），你会发现它像一位固执的老刑警，翻来覆去只说同一句话——“我怀疑是你，就是你，肯定是你”。生成 50 个 Token 里，30 个是重复的 n-gram。它掉进了“复读机诅咒”。
>
> 如果反过来，完全随机选（**高 Temperature**），它又像一位思维过于发散的实习生——上一句还在分析指纹，下一句突然讨论晚餐吃什么。概率分布被 Temperature 抹平后，所有 Token 的机会均等，输出的文本完全失控。
>
> **Top-k** 试图在两者之间找平衡：只从概率最高的前 k 个里选。但 `k=50` 是暴力截断——如果前 50 个只覆盖了 60% 的概率质量，剩余的 40% 被硬生生抛弃（截断损失）。
>
> **Top-p（核采样）** 是更优雅的方案：从概率最高的 Token 开始累加，直到累计概率达到 `p=0.9`。候选集随上下文动态变化——信息量大时候选集宽，信息量小时候选集窄。这就是当前所有 SOTA Agent 的默认“性格开关”。
>
> 而为了防止 Agent 在长思考（CoT）中车轱辘话来回说，我们还要在 logits 层装一道 **惩罚机制（Repetition Penalty）**：对已经说过的 Token 做减法，强迫它换话题。
>
> 本章没有“正确答案”——只有“合适的选择”。你将亲手操控这些旋钮，观察同一个 Prompt 在四种策略下分裂出四种截然不同的“人格”。而你将学会的，是如何为你的 Agent 选择最匹配的那一个。

- **冻结 Embedding/底层（Freeze）**：只训最后几层，防止灾难性遗忘
- **LoRA**（链接第 9 章 SVD）：用两个低秩矩阵 `(D×r)` 和 `(r×D)` 模拟 `ΔW`，可训练参数量减少 99%
- 代码：用 `peft` 库的 `LoraConfig` 注入 LoRA，对比三种策略的 `trainable_params` 数量

  34.4　影评情感分类实战（替换输出层为分类头）

- 加载预训练 GPT-2 → 替换 `lm_head` 为 2 分类头 → 训练 1 个 Epoch
- 绘制：训练 Loss、验证 Acc、以及 **PPL 的联合变化曲线**（链接 19.6 节，微调后 PPL 骤降说明模型变“确定”了）

  34.5　过拟合的“心电图”与 Early Stopping

- 验证集 Acc 上升但 Loss 上升 → 过拟合信号
- 引入 Early Stopping 和 Weight Decay（链接 24.4 节 AdamW）

## 第35章　Agent 结构化的”紧箍咒” —— 工具调用与格式约束

> **本章目标**：把第 22.5 节的 Logit Bias 扩展成一套完整的“Agent 格式保险箱”工程方案。

> 🕵️ **开篇导语：案件报告的“标准格式”**
>
> Agent 学成了。它拥有通识知识（预训练），掌握了专属技能（微调），说话风格也被你调教得恰到好处（采样策略）。现在，你把它派上了真正的战场——**自动调用外部工具**。查天气、搜新闻、发送邮件、查询数据库……
>
> 但你很快发现一个令人崩溃的现实：**模型什么都懂，就是不好好写报告。**
>
> 你让它调用 `search` 工具，它返回 `{"action": "search", "query": "天气"}` ——完美。下一轮，同样的输入，它返回 `Search query is 天气`（丢了 JSON 外壳）。再下一轮，它返回 `{action: search, query: "天气"}`（少了引号，`json.loads()` 直接报错）。Agent 的“创造力”在工具调用场景下变成了灾难——每一次格式错误都意味着重试、延迟、甚至执行失败。
>
> 你需要一套 **三层的“紧箍咒”格式保险箱**：
>
> - **第一层（Prompt 约束）**：在系统指令里写死 `"You MUST output valid JSON"`，靠模型的“服从性”；
> - **第二层（Logit Bias 硬约束）**：在第 22 章的基础上，在每一步生成时用 `torch.where` 将非法 Token 的 logits 直接钉死在 `-inf`，让模型**根本说不出**不合法的字符；
> - **第三层（解析重试）**：万一前两层失守，用低温度（`T=0.1`）重新采样一次，附指数退避机制——让模型“冷静下来”重新写。
>
> 本章你将亲手量化：温度从 0.1 提升到 1.0 时，JSON 解析失败率如何从接近 0% 飙升至 40% 以上——然后用你的三层保险把失败率压回安全区。你还会接触到 Agent 结构化的终极武器：**Guidance / Outlines**，用上下文无关文法在解码时强制匹配正则表达式——那是生产级 Agent 的真实选择。

35.1　工具调用（Function Calling）的协议

- 定义 JSON Schema：`{"name": "search", "parameters": {"query": {"type": "string"}}}`
- 将 Schema 注入 System Prompt，强制模型输出可解析的字符串
- 代码：用 `json.dumps` 将 Schema 序列化并拼入 Prompt

  35.2　JSON 模式的三层保险

- **第一层（Prompt 约束）**：在 System Prompt 中写死 `"You MUST output valid JSON"`
- **第二层（Logit Bias 硬约束——教学简化版）**：在每一步用 `torch.where` 将非法 token 的 logits 置为 `-inf`。本实现只限制特殊 Token（`{`、`"`、`:`、`}`）在关键位置出现，不做完整状态机（完整实现见 35.4 的生产级方案）
- **第三层（解析重试）**：若 `json.loads()` 失败，用低温度（`T=0.1`）重新采样一次，附指数退避重试机制
- 代码：实现 `ensure_json` 装饰器，自动检测输出并触发重试

  35.3　温度与格式错误率的量化实验

- 对同一个 API 调用 Prompt，在 `T=0.1、0.5、1.0` 下各跑 200 次
- 统计 JSON 解析失败率，画出 `温度-错误率` 柱状图
- 结论：低温度（0.1~0.3）是工具调用的“安全区”

  35.4　结构化生成的终极武器（选读）

- 介绍 **Guidance / Outlines** 库的直觉：用上下文无关文法（CFG）在解码时强制匹配正则表达式
- 代码：用 `outlines` 库一行代码实现 JSON 模式生成
